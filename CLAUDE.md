# CLAUDE.md — Kinda プロジェクト 憲法

> このファイルは「Kinda とは何か」を定義する**不変の憲法**です。
> AIアシスタント（Claude / Claude Code 等）は、毎セッションこのファイルを最初に読み、
> ここに書かれた世界観・トーン・ルールを前提として作業すること。

> **重要：このファイルに日々の作業ログを追記しないこと。**

## 1. Kinda とは何か（最重要・ここを外すと全部ズレる）

**Kinda（キンダ）** — 結婚相談所を「カウンセラー個人の口コミ」で選んでから予約できる、口コミ・レビュー型プラットフォーム。@cosme や食べログの「結婚相談所版」と考えると正確。

- **業態**：婚活支援産業。結婚相談所の掲載プラットフォーム。
- **メインターゲット**：結婚相談所を探している**未入会者**。加えて、入会済み・交際中の人、LGBTQ+ を含む「関係を築いている最中のすべての人」。
- **核心の価値**：面談した人だけが書ける口コミ。広告に頼らない**ユーザーファースト**な設計。
- **外部発信での自称**：「ユーザーファーストな口コミ・レビュー型プラットフォーム」。
  - ⚠️ **「中立」という言葉は使わない**（業態を否定する表現になるため）。

**ビジョン：**
世の中のレビューサイトは「関係が出来上がった人」のためにある。
Kinda は「今まさに関係を作っているふたり」のためにある。
婚活準備 → お見合い → デート → プロポーズ → その先まで拡張していく。

## 2. 「婚活と言わない」の正確な線引き（誤解多発・必読）

「婚活という言葉を避ける」は**業態の否定ではなく、サイトのトーン上の選択**。場所によって扱いが違う。

| 場所 | 「婚活 / 結婚相談所 / お見合い」の扱い |
|---|---|
| ヒーロー・CTA・ブランドコピー | **使わない**。「好きな人」「ふたり」「過ごす日々」で書く |
| 診断ページ・天気ページ | **中立語彙のみ**。婚活ワードを出さない |
| コラム記事の本文 | **使ってOK**（SEO上むしろ必須）。ただし主役にしない |

- コラムは「婚活 / 結婚相談所」を**SEOキーワードとして意図的に取りに行く**。
- ただしコラム本文でも「結婚」「成婚」をゴールとして煽らない。「好きな人」「ふたり」「過ごす日々」を中核に置く。

## 3. ブランドトーン（全コンテンツ共通）

| ルール | 詳細 |
|---|---|
| **焦らせない・比較しない** | 「早くしないと」「周りは」で追い詰めない。「自分のペース」を肯定する |
| **空虚な共感を禁止** | 「わかってるよ」「すごい」のような中身のない共感はしない。実用的な価値を返す |
| **LGBTQ+ 包摂** | 「異性」「男女」を当たり前にしない。「相手」「ふたり」で書く |
| **詩的トーン** | 天気のメタファーを軸に、断定より寄り添い |
| **絵文字なし** | コンテンツ・UI 文言で絵文字を一切使わない |
| **侮辱的に読まれうる表現を避ける** | タイプ名・診断結果などで、相手を下に見る表現を使わない |

Kinda は「カウンセラーの代替」ではなく、**気持ちを整理して渡す中立的なファシリテーター／書類整理ツール**として振る舞う。

### 絵文字の細則（2026-06-08 追記）
- サイト本文・記事・UI 文言：**絵文字なし**（従来どおり厳守）。
- 外部SNS（note / Instagram / X 等）のキャプション：絵文字を**最小限・区切り用途で許容**。

### 著者・アバター・声の方針（2026-06-08 追記）
- 実名「**ふうか**」を著者として用いる（E-E-A-T）。
- **顔出しはしない**。著者アイコン／登場ビジュアルは「**クレイで作ったふうか像**」を用いる。
- **音声（ふうかの声）は利用可**：note 音声記事、リールのナレーション等（顔は出さない）。

## 4. ビジュアル定義（フロントサイト＝ユーザー向け）

**ミニチュアクレイ風**の独自ビジュアルが差別化資産。

| 項目 | 値 |
|---|---|
| 質感 | ミニチュアクレイ（粘土）風。柔らかいマット質感、丸み、やわらかい影 |
| 背景色 | `#F5EEE6`（ウォームベージュ） |
| アクセント | `#D4A090`（ダスティローズ／テラコッタ系） |
| 雰囲気 | 温かい・穏やか・癒し。差し込む光は上から柔らかく |

- 画像は ChatGPT（DALL-E）で生成。生成後は**必ず WebP 変換**し、ファーストビュー画像には `next/image` の `priority` を付ける（LCP対策）。
- ⚠️ **旧デザイン（ゴールド #C8A97A・「スタイリッシュ／高級感」路線）はフロントサイトでは使わない**。クレイ風ベージュが正。

> 注：admin（`futarive-admin`）/ counselor 向け管理画面は、別のUI方針（実務寄り）で構わない。
> クレイ風ベージュは**ユーザー向けフロントサイトのブランド定義**。管理画面まで縛らない。

## 5. セクション語彙（Kinda の構成要素と役割）

| セクション | 役割 |
|---|---|
| **Kinda note** | 気持ちの整理（診断問診票）。整理した内容をそのままカウンセラーに渡せる |
| **Kinda type** | カウンセラーマッチング診断。診断するだけで合うカウンセラーが見つかる（送客メイン導線） |
| **Kinda talk** | カウンセラーに直接会う（対応相談所のみ） |
| **Kinda act** | お見合い・デートに使いやすい場所のレコメンド |
| **Kinda glow** | 好きな人に会う前に、自分を整える時間 |
| **Kinda story** | 先に進んだ人の物語（送客目的ではなく控えめに） |

- LGBTQ+ は **Kinda act / glow / note** で包摂する。**Kinda talk は対応相談所のみ**。

### Kinda story の細則（2026-06-08 追記）
- Story は実在の人物が題材。**項目別の掲載同意（Web／SNS二次利用／写真／実名・仮名／相談所名／相手配慮）がない素材は下書きも作らない。**
- 既定は「**仮名・年代ぼかし・クレイ情景のみ（実写なし）**」。同意が取れた項目だけ開放。
- 構造化データは **Article のみ**（Review/Rating は付けない＝ステマ規制・広告誤認回避）。
- 自社関与（Emma 等）の事例は**関係性を明示する一文**を入れる。
- 撤回依頼には速やかに応じる（`is_published=false` ＋ 同意取得日・範囲を記録）。
- **Story は編集ゲート制を維持し、公開投稿型（オープンUGC）にはしない。** やらせ排除（面談した人だけの口コミ）という信頼の核を守るため。「ひとりじゃない」という belonging は Story ではなく **Kinda Note 側**で実現する（TODO のロードマップ参照）。

#### Story 実装の不変則（2026-06-27 確定）
- **Story の実体ルートは `/kinda-story`**（`src/lib/mock/stories.ts` のリスト方式）。MDX 方式（`/story`）は採用しない。新規投稿は `STORIES` 配列にオブジェクトを 1 件足す。具体手順は TODO 参照。
- **クレイ画像は `stage` から自動出し分け**（成婚=seikon／交際中=kosai／活動中=katsudo）。記事ごとの画像生成・個別指定は不要。
- **構造化データは Article + FAQPage のみ**（Review/Rating/AggregateRating は不可）。
- **相談所名が同意範囲外の Story は、コードにも実名を持たせない**（mock はクライアントへ配信されるため）。著者カードを出さず、自社関与なら透明性の一文を本文に入れる。

### Tailwind CSS v4 注意（v3と違う）
- `tailwind.config.ts` は不要。`globals.css` の `@theme inline` で設定する。

- クライアント：`createBrowserClient` / サーバー：`createServerClient`（cookies連携）。

- ⚠️ テーブル定義の確認は**マイグレーションファイルより `list_tables`（Supabase MCP）を信頼**する。

### 開発環境の制約
- 開発は iPad Pro / iPhone から Claude Code 経由。**`localhost` での確認は不可**。
- 視覚確認の主手段は **Vercel デプロイ（プレビュー / Production alias）**。

## 8. 重要なビジネスルール

### 口コミの信頼性
- **相談所口コミ**：Kinda 経由で面談完了した人のみ投稿可（面談完了後に専用URL＋認証コードを発行）。
- **お店口コミ**：利用後なら誰でも投稿可。
- **代理掲載口コミ**：管理画面から運営が入力。必ず `source_type='proxy'` ＋「**代理掲載**」バッジを表示（景品表示法・ステルスマーケティング規制対応）。

### 収益モデル
- **送客料：¥5,000 / 1件（成果報酬のみ・初期費用なし）。**
- 初期は無料掲載＋成果報酬モデルで展開。将来的にプレミアム掲載オプションの追加を検討。
- 国内 4,000〜5,000社の結婚相談所のうち約2,500社を3年以内にターゲット。

### 設計上の原則
- **ユーザーファーストが最優先**（収益最大化より上）。
- Admin パネル（`futarive-admin`）はユーザー向けサイトから**完全分離**（データ露出リスクの排除＝意図的判断）。

## 10. ブランチ運用（2026-05-23 更新）

> **重要：CLAUDE.md / TODO.md / WORKLOG.md 内に登場する過去のブランチ名（uDUoW 等）は履歴情報であり、現在の作業ブランチ指定ではない。新規セッション開始時はこの §10 を信頼すること。**

### 原則

- **新規の機能追加・修正は `main` から派生した short-lived feature branch で行う → PR → main にマージ → ブランチ削除**
- 過去の長期ブランチ（`claude/implement-kinda-talk-uDUoW` 等）は歴史的経緯のもので、**新規作業に使わない**
- ドキュメント（CLAUDE.md / TODO.md / WORKLOG.md）の更新は main 直接 push 可。機能コード変更は PR 経由
- 各 Vercel project は独立した Production Branch を持つ。下表の通り

### 各 Vercel project の Production Branch（現行スナップショット）

| Vercel project | Production Branch | 外部公開 URL |
|---|---|---|
| my-app-rp9u（ユーザーサイト）| `main` | https://kinda.jp （旧：https://my-app-rp9u.vercel.app）|
| futarive-counselor（カウンセラー管理画面）| `claude/fix-profile-creation-1clpG` | https://futarive-counselor.vercel.app |
| futarive-admin（管理画面）| `main`（実体は preview 運用）| `claude/futarive-admin-dashboard-iKBfw` の最新 preview URL を Vercel ダッシュボードで都度確認 |

> ドメイン：2026-06-05 に **`kinda.jp`** を取得（お名前.com・法人名義 AGOGLIFE Inc.・初年度0円・NS は `ns1/ns2.vercel-dns.com`）。`my-app-rp9u` に `kinda.jp` / `www.kinda.jp` を登録。本番 URL は **`kinda.jp` に統一**（`my-app-rp9u.vercel.app` も引き続き有効）。⚠️ コードの `metadataBase` は旧 `www.kinda-futari.app` のままなので要修正（OGP/canonical 用・TODO 参照）。

admin は user 一人運用 + カスタムドメイン不要のため production target は使わず preview URL を実体運用。将来複数人運用になったら counselor と同じ手順（Production Branch 切替 + Deployment Protection Disabled）を実施する。

### 作業対象ごとのブランチ選択

| 修正対象 | 派生元 | 反映先 URL |
|---|---|---|
| ユーザーサイト（`src/`）| `main` → feature branch → PR → main | https://kinda.jp |
| counselor 管理画面（`futarive-counselor/`）| `claude/fix-profile-creation-1clpG` に直接 push、または feature branch → PR | https://futarive-counselor.vercel.app |

### 注意

- Vercel の Ignored Build Step が古いブランチ名で固定されている可能性あり。新しいブランチ名で push しても build が走らないときは Vercel ダッシュボードで Ignored Build Step を確認・更新する
- `.vercelignore` は repo root から全 Vercel project に適用される。`futarive-counselor/` `futarive-admin/` を ignore しているのは my-app build がサブアプリのファイルを拾わないため。サブアプリ project は Root Directory 設定 + 上記の Production Branch 設定で運用

### デプロイ前の必須チェック（2026-06-01 追加・厳守）

> 過去にユーザーサイト作業を誤ったブランチ起点で始める / ローカル `main` を信用する事故あり。再発防止。

- **ローカル `main` は信用しない。常に `origin/main` を真とする**（ローカル main が unrelated history の古い commit を指すことがある）。作業前に `git fetch origin main` → `git checkout -B <feature> origin/main` で **origin/main 起点**にする。
- ユーザーサイト(`src/`)系統と counselor(`futarive-counselor/`)系統は**履歴が分岐している**。対象に応じて正しい系統を起点にし、取り違えない。
- **新しいセッションでユーザーからデプロイを依頼されたら、Claude は実装に着手する前に「前回使ったブランチ名と最新コミット（`origin/main` の HEAD 等）」をユーザーに伝え、起点ブランチの確認を取ること。** 確認なしに作業を始めない。

## 11. Kinda voices 運用フロー（取材→記事化）

> Kinda voices はカウンセラー個人単位の取材記事セクション。
> 掲載初期の信頼担保・SEO・営業ツールの3役を兼ねる。
> このセクションは毎回の取材作業の標準手順として参照すること。

### 記事の基本設計

| 項目 | 内容 |
|---|---|
| 単位 | カウンセラー個人単位（1相談所1人から始める） |
| 推奨文字数 | 3,000〜4,500字 |
| 推奨画像 | 5〜8枚（取材写真） |
| 主要SEOキーワード | 相談所名＋カウンセラー名＋地域名をtitleに |
| 内部リンク | 記事末尾 → カウンセラー詳細ページ |
| CTA文言 | 「〇〇さんの話を、直接聞いてみる。」（緊急感・比較禁止） |

### 記事構成（8セクション・固定）

1. タイトル＋ヘッダー画像
2. リード文（200字以内・核心を先に）
3. プロフィールサマリー（箱組み）
4. この仕事を選んだ理由（原体験）
5. どんな人を担当してきたか
6. 相談所の空気感・働き方
7. 印象に残っているひとこと（引用ブロック・SNS拡散設計）
8. クロージング＋CTA → カウンセラー詳細ページへの内部リンク

### Kindaトーン（Kinda voices記事専用の注意）

- 励まさない・断言しない・現象描写にとどめる（Kinda note と同じ原則）
- 「婚活」「結婚」はSEO上本文で使ってOK。ただしゴールとして煽らない
- 「婚活しなくちゃ」より「誰かと過ごしたい」に寄り添う視点で書く
- 引用ブロックには「業界の常識をそっと外すような言葉」を選ぶ

### 取材→記事化ワークフロー（MacBook Neo環境）

```
① iPhoneのボイスメモで取材を録音
② AirDropでMacのデスクトップに転送
③ ターミナルで以下を実行
   whisper ~/Desktop/ファイル名.m4a --language ja --model medium
   （初回のみmediumモデルDL・約5分）
④ デスクトップに .txt ファイルが自動生成される
⑤ Claude.aiのチャットに貼り付けて「Kinda voicesの構成で記事にして」と指示
⑥ 完成原稿をMDXファイルとしてKinda voicesに追加
```

### Whisper環境（MacBook Neo・設定済み）

- Homebrew・ffmpeg・openai-whisper（Python 3.9）インストール済み
- PATH設定済み（~/.zprofile）。費用：完全無料（ローカル処理）

## 12. 決済・メール基盤（Stripe / Resend）

> 2026-06-04 確定。実装指示書は `docs/implementation/claude-code-stripe-resend-implementation.md` を正とする。

### Stripe（決済）

- 支払い手段：**クレジットカード登録制のみ**（銀行振込は対応しない・例外なし）。カード未登録の相談所は契約不可として断る。
- 課金タイミング：**予約が確定した時点で即時 ¥5,000**（`off_session` 課金）。※旧「予約成立＋24時間猶予」モデルは廃止（2026-06-05 変更）。
- 使用機能：Payment Intents + Customers + Webhook + Refund API。Billing・Invoicing は不使用（手数料3.6%のみ）。
- 決済成功Webhook（`payment_intent.succeeded`）受信 → 予約 `confirmed` ＋ ユーザー連絡先を相談所管理画面に自動開示。
- Webhook処理のみ `service_role_key` を使用。それ以外は RLS を必ず有効に。

### 返金ルール（2026-06-05 改定・現行）

> 旧「24時間以内は全額返金」モデルから変更。**予約確定で即時課金・以後は原則返金なし**、例外は運営事務局が個別相談で判断（運用しながら相談所の声で調整）。
> ⚠️ この新ルールに合わせ、契約書(`docs/contracts/agency-agreement.md`)・事業者向け特商法(`tokushoho-agency-b2b.md`)・営業デッキ返金脚注・Stripe実装指示書を**要同期**（顧問弁護士レビュー前提・TODO 管理）。

| ケース | 処理 |
|---|---|
| 予約確定 | 即時 ¥5,000 課金（猶予なし） |
| 以後のキャンセル・当日キャンセル・ユーザー不参加（ドタキャン） | **原則返金なし**（課金確定） |
| やむを得ない事情 | 運営事務局へ相談 → **個別判断で返金しうる**（相談ベース） |

- 相談所⇔ユーザー（会員）間のトラブル（ドタキャンによる交通費・店舗代等を含む）に Kinda は責任を負わない（agency 利用規約 第14条で担保済み）。
