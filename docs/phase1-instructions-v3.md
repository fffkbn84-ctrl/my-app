## このフェーズの目的

現在の Kinda note コンポーネントを、新しい設計方針に基づいて全面的に書き直す。
**このフェーズではDBには一切触らない。** ゲストでの利用のみ対応し、履歴はlocalStorageに一時保存する。

## 重要な設計原則

以下の原則を**すべての実装判断の上位**に置くこと。矛盾が生じた場合はこの原則を優先する。

### 原則1：Kindaは診断ツールではなく「翻訳ツール」
- ユーザーの言葉にならない気持ちを、カウンセラーに渡せる形に翻訳する
- 「答えを出す」のではなく「対話のテーブルに着かせる」ことが目的

### 原則2：カウンセラーを評価しない
- 「合わない」「変えるべき」を言わない
- 担当変更の選択肢を示唆しない
- Kinda type（カウンセラーとの相性チェック）への誘導は**入会前ルートのみ**

### 原則3：ユーザーの判断を奪わない
- 退会・契約変更等をKindaが勧めない
- 「まずカウンセラーに話してみる」を第一選択肢にする
- それでも解決しない場合の次のアクションはユーザー自身が決める

### 原則4：広告・特定事業者の推奨をしない
- 特定の相談所の料金を推奨・非推奨しない
- 「多くの相談所では〜」「〜の場所が多い」という表現を徹底する
- 「成婚」の定義は相談所ごとに違うことを明記し、詳細に踏み込まない

### 原則5：諦めシグナルに「カウンセラー変更」を返さない

ユーザーの回答や自由記述に「相談しても変わらない」「伝わらない」「カウンセラーが頼りない」等の**諦めシグナル**が検知されても、Kinda は**カウンセラー変更を示唆しない**。代わりに以下の対応を取る。

- 「メモを見せてみて」で留める。つまり、**まず今のカウンセラーに一度は見せる**ことを提案する。
- カウンセラーとの関係を**終わらせる方向に動かさない**。
- それでも状況が変わらない場合の次のアクションは、ユーザー自身が決めること（Kinda はそこに踏み込まない）。

**理由：** 担当変更は相談所内部の重い手続きで、Kinda が軽々に示唆すべきではない。また、「変わらない」と感じているのはカウンセラー側の問題ではなく、ユーザーが自分の気持ちをまだ言語化できていないケースが大半で、メモを渡すことで関係が動くことが多い。

### 原則6：ネガティブ2つ以上でポジティブ判定をオーバーライド

ユーザーの回答で違和感・疲労・迷い・億劫さなどの**ネガティブシグナルが2つ以上**検知された場合、たとえポジティブ選択肢（「また会いたい」「好き」等）が選ばれていても、ポジティブ系タイプ（晴れ間・朝焼け等）への分岐を**オーバーライド**して、中立〜ネガティブ系タイプに振り分ける。

**理由：** ユーザーが「また会いたい」「好き」を選んでいても、同時に複数のネガティブシグナルを出している場合、それは自信のなさや社会的プレッシャーから来ている可能性が高い。そのままポジティブ系として扱うと、ユーザーが感じている違和感を Kinda が打ち消してしまう。ネガティブオーバーライドは、**ユーザーの違和感を正当化する**ための仕組み。

**実装は各ルートの分岐ロジック内で行う。** omiai / date1 / kousai ルートそれぞれの `decideXxxType` 関数を参照。

## DBに触らない範囲での localStorage 設計

Kinda note 完了時に以下の形式で保存する：

```typescript
// localStorage key: "kinda_note_history"
type KindaNoteHistoryItem = {
  id: string;              // uuid v4
  route: "pre" | "waiting" | "omiai" | "date1" | "kousai" | "multiple";
  result_type: string;     // 例: "Kinda 朝もや"
  weather: string;         // 例: "morning_mist"
  answers: Record<string, any>;  // 回答全体
  created_at: string;      // ISO 8601
};

type KindaNoteHistory = KindaNoteHistoryItem[];
```

最大100件まで保存（古いものから削除）。

**重要**：この構造は将来のDBスキーマ（`kinda_note_sessions` テーブル）と同じ形にしておくこと。フェーズ4でDB移行する時に変換ロジックを書かずに済むようにする。

## encourage の扱い（全ルート共通ルール）

**encourage は pre ルートのみで使用する。**

### 背景

Kinda は「翻訳ツール」であり、ユーザーに余計なメッセージを重ねることは設計原則に反する（原則3：ユーザーの判断を奪わない）。以前は全タイプに encourage（背中を押す一言）を付けていたが、以下の理由で方針を変更する。

- **pre ルート**：入会前の人には「踏み出してみる」ための後押しが必要。コンバージョン目的で encourage を**残す**。
- **waiting / omiai / date1 / kousai / multiple**：既に活動中の人には、Kinda が判断や気持ちを誘導すべきではない。結果を読んだユーザーが自分でどう動くかを決める。encourage は**削除**し、透明なツールであることを優先する。

### 実装ルール

- pre ルートの3タイプ（朝もや / 夜明け前 / 花曇り）には `encourage` フィールドを残す。
- それ以外のルートのタイプデータから `encourage` フィールドを**削除**する。
- 結果画面のレンダリング側も、encourage が存在しない場合は該当セクション自体を描画しないようにする（空のカードや区切り線を残さない）。

## pre ルートのタイプ分岐ロジック

```typescript
function decidePreType(answers) {
  const reasons = answers.q2 || [];
  
  const infoReasons = [
    "相場がわからない（高いのか安いのかも判断できない）",
    "毎月の費用を続けられるか不安",
    "払ったのに結果が出なかったら、と思うと踏み出せない",
    "どんな人が活動しているか想像できない",
    "相談所に入ることへの抵抗感がある",
  ];
  const confidenceReasons = [
    "自分に合うカウンセラーがいるか不安",
    "本当に相手が見つかるか自信がない",
  ];
  
  const infoCount = reasons.filter(r => infoReasons.includes(r)).length;
  const confCount = reasons.filter(r => confidenceReasons.includes(r)).length;
  
  // 優先順位：自信不足 > 情報不足 > 慎重型
  if (confCount >= 2) return "yoakemae";          // 夜明け前
  if (infoCount >= 2 && confCount <= 1) return "asamoya";  // 朝もや
  return "hanagumori";                             // 花曇り
}
```

## pre タイプ1：Kinda 朝もや

- **weather**: `morning_mist`
- **color**: `C.green` 系（既存の緑系を使用）

### 天気の解説

やわらかな霧が世界の輪郭をほどく朝。
すべてがまだ決まっていない、その白さの中にあなたはいます。

### 第1層（常に表示）
「踏み出せない」のは、慎重さじゃなくて**情報が足りていないだけ**かもしれない。結婚相談所って普段の会話で話題に出るものじゃないから、仕組みや相場が見えないのは当たり前のことだよ。見えないものには、誰だって一歩が出にくい。

### 第2層（選択項目ごとに条件表示）

**「相場がわからない」選択時：**
結婚相談所の月額は、**1.5〜3.5万円**が中心。ジムやサブスクと同じくらいの感覚の人が多いよ。ただし初期費用や成婚料は相談所ごとに幅があるから、比較するときは月額だけじゃなく**総額**で見るのがおすすめ。

**「毎月の費用を続けられるか不安」選択時：**
続けられるかどうか、今ここで一人で判断しなくて大丈夫。ほとんどの相談所は**初回面談が無料**で、料金プランや支払い方法を詳しく聞ける。聞いてから「続けられそうか」を決めれば十分だよ。

**「払ったのに結果が出なかったら」選択時：**
結果が出なかったらどうしよう、という不安はごく当たり前の気持ちだよ。多くの相談所には「**成婚料**」という仕組みがあって、これは一緒にいたいと思える相手が見つかった時に発生する料金のこと。ただし「成婚」の定義は相談所ごとに違うから、契約前にそこで聞くのが確実だよ。退会条件も一緒に確認しておくと安心。

**「どんな人が活動しているか想像できない」選択時：**
活動している人の大半は、**ごく普通の会社員や専門職**。特別に社交的な人ばかりでも、婚活に必死な人ばかりでもないよ。「自然な出会いがなかった」という理由でいる人が一番多い。

**「相談所に入ることへの抵抗感がある」選択時：**
その抵抗感にはちゃんと意味がある。「自分で見つけられなかった」という気持ちとセットになっていることが多いから。でも**今は3人に1人が何らかのマッチングサービスで出会ってる時代**。相談所も、選択肢のひとつとして普通になってきてるよ。

### 第3層（組み合わせ条件で動的生成）

**情報系選択が3つ以上：**
これだけ多くの「わからない」を抱えてるってことは、一人で調べるには限界がきてるサインかも。面談は情報収集の場として使って全然いいよ。契約しに行く場じゃない。

**「相場」+「続けられるか」両方選択：**
お金の不安が重なっているのは、きっと「失敗したくない」という真剣さの裏返し。無料面談で**総額と支払いパターン**まで聞いてしまえば、この2つの不安はまとめて解決できるよ。

**「抵抗感」+「どんな人」両方選択：**
「相談所の世界」そのものがまだ遠く感じているんだね。そういう時は、口コミで**実際にそこに行った人の言葉**を読むだけでも距離が縮まる。

### 天気の解説

## タイプ分岐ロジック

```typescript
function decideOmiaiType(answers) {
  const meetAgain = answers.q3;
  const impressions = answers.q4_impressions || [];
  
  const positives = [
    "思ってたより話しやすかった",
    "会話がすごく盛り上がった",
    "ドキドキした瞬間があった",
    "なんか良さそうだった",
  ];
  const negatives = [
    "見た目が思ってたのと少し違った",
    "沈黙が気になった",
    "会話が合わないと感じた",
    "違和感があった",
  ];
  const hasPositive = impressions.some(i => positives.includes(i));
  const negativeCount = impressions.filter(i => negatives.includes(i)).length;
  
  // 【重要】ネガティブ2つ以上の場合、ポジティブ判定をオーバーライド
  // 「また会いたい」と答えていても、違和感・沈黙・合わなさが複数あるなら、
  // 晴れ間ではなく天使の梯子または風の日へ振り分ける
  if (negativeCount >= 2) {
    if (meetAgain === "迷っている" || meetAgain === "もう一度なら会えそう") {
      return "angels_ladder";
    }
    if (meetAgain === "次に進むのは難しい") return "kazenohi";
    // 「また会いたい」でもネガ2つ以上なら天使の梯子に寄せる
    return "angels_ladder";
  }
  
  if (meetAgain === "また会いたい" || hasPositive) return "hareme";       // 晴れ間
  if (meetAgain === "迷っている" || meetAgain === "もう一度なら会えそう") return "angels_ladder";  // 天使の梯子
  return "kazenohi";  // 風の日
}
```

**ロジックの設計意図：**

ネガティブシグナル（違和感・沈黙・会話の合わなさ・見た目ギャップ）が2つ以上重なっている状態で、ユーザーが「また会いたい」と選んでいる場合、それは**自信のなさや社会的プレッシャー**から来ている可能性が高い。そのまま晴れ間（ポジティブ系）として扱うと、ユーザーが感じている違和感を Kinda が打ち消してしまうことになる。ネガティブオーバーライドを入れることで、**ユーザーの違和感を正当化する**方向に判定を寄せる（その場合は天使の梯子または風の日へ振り分ける）。

## omiai タイプ2：Kinda 天使の梯子

### 天気の解説

雲の隙間から、幾筋もの光が静かに降りる現象。
確信ではなく予感のような、そのやわらかな光の中にあなたはいます。

「また会いたい」でも「会いたくない」でもなく、間にいる。これは**判断材料が足りていない状態**で、お見合いの短い時間だけで結論を出すのは難しいから、むしろ自然な感覚だよ。「迷える」ということは、相手に対して何かしらの引っかかりが——良い意味でも悪い意味でも——ちゃんと生まれているってこと。無味乾燥だったら、迷うことすらできない。

**「ときめきはなかったけど悪くなかった」：** この感覚、**実は軽く見ない方がいい**。「悪くなかった」は落ち着いた関係のスタート地点になりやすくて、ときめきは後から育つこともある。逆にときめきだけで始まった関係より続く傾向すらあるよ。

function decideKousaiType(answers) {
  const worries = answers.q3 || [];
  const atmosphere = answers.q2;
  
  // 各シグナルを独立フラグとして定義
  const hasPositive = worries.some(w => 
    w.includes("好き") || w.includes("次のステップ")
  );

                    worries.includes("会話が盛り上がらないことがある");
  // 「疲れ」：心身の消耗（億劫とは区別）
  const hasTired = worries.some(w =>

  );
  // 「迷い」：気持ちがはっきりしない

                   worries.includes("自分の気持ちがよくわからない");
  
  // ネガティブシグナルの総数（positive判定オーバーライド用）
  const negativeCount = [hasOukkyuu, hasIwakan, hasTired, hasMayoi]
    .filter(Boolean).length;

  if (hasTired) return "shizukana_kumori";                        // 静かな曇り
  return "kiri";                                                   // 霧
}

# 天気アイコンの実装（フェーズ1暫定）

フェーズ1では**SVGでプレースホルダー**を作る。後から画像差し替えができる構造にしておくこと。

```tsx
// components/WeatherIcon.tsx
type WeatherKey = 
  | "morning_mist" | "pre_dawn" | "flower_overcast"          // pre
  | "light_rain_start" | "light_rain" | "rain_cloud" | "thunderstorm"  // waiting
  | "sun_break" | "angels_ladder" | "windy_day"              // omiai
  | "light_sunrise" | "wandering_clouds" | "cold_wind"       // date1
  | "sunrise" | "dissonance_wind" | "quiet_overcast" | "mist"  // kousai
  | "twilight" | "windy_sunshine" | "faint_sunlight";        // multiple

export function WeatherIcon({ weather, size = 80 }: { weather: WeatherKey; size?: number }) {
  // 将来画像に差し替える時のために、パスベースで分岐できる構造に
  const imagePath = `/images/weather/${weather}.webp`;
  
  // フェーズ1：SVGで暫定実装
  return <WeatherSvg weather={weather} size={size} />;
  
  // フェーズ2以降：画像に差し替え
  // return <img src={imagePath} width={size} height={size} alt="" />;
}
```

SVGの実装は、各天気のコンセプトに沿ってシンプルに作る：
- morning_mist：もやの中に薄日（楕円のグラデーション）
- pre_dawn：地平線と星1つ
- flower_overcast：雲と花びら数枚
- light_rain_start〜thunderstorm：雨粒の数と雷の有無で段階表現
- sun_break：雲の切れ目から光
- その他はコンセプトに沿って線画ベースで

色は既存の `C` オブジェクトのカラーパレットを使用し、**outline + soft fill** のスタイルで統一。

# ボタン構成

## 結果画面のボタン（ルート別）

### pre ルート（入会前）

pre ルートの全タイプ（朝もや / 夜明け前 / 花曇り）で、以下の5ボタンを表示する。

```tsx
<button primary>Kinda type であなたに合うカウンセラーを</button>
<button secondary>Kinda talk でカウンセラーを見てみる</button>
<button secondary>Kinda glow で自分を整える時間を</button>
<button tertiary>Kinda story を見てみる</button>
<button ghost>もう一度やってみる</button>
```

**ボタン序列（視覚的な大きさ）：**

1. **primary**（最大、`C.accent` 塗り、白文字、pill型）：Kinda type ← 送客メイン
2. **secondary**（中、`C.card` 塗り、border あり）：Kinda talk
3. **secondary**（中、`C.card` 塗り、border あり）：Kinda glow
4. **tertiary**（小、border のみ、細め）：Kinda story
5. **ghost**（最小、テキストのみ）：もう一度やってみる

**設計意図：**

- Kinda glow（自分を整える時間）は疲れ系タイプだけでなく**pre ルート全タイプで表示**する。入会前の段階では「まず自分のコンディションを整える」選択肢を常に提示したい。
- Kinda story は送客目的ではなく、先に進んだ人の物語に触れる機会として、控えめな位置に置く。

### waiting / omiai / date1 / kousai / multiple ルート

活動中ルートでは、Kinda は「翻訳ツール」として機能する。ボタン文言も、機能的な表現ではなく情緒的な表現にする。

```tsx
<button primary>この気持ちを、そのまま渡す</button>
<button secondary>画像にして持っておく</button>
<button ghost>もう一度やってみる</button>

{/* ─── 他の機能も見てみる（活動中ルート全タイプで表示） ─── */}
<div className="sub-links" style={{
  marginTop: 32,
  paddingTop: 20,
  borderTop: `1px solid ${C.border}`,
  textAlign: "center",
  opacity: 0.85,
}}>
  <p style={{ fontSize: 12, color: C.textMuted, marginBottom: 12 }}>
    他の機能も見てみる
  </p>
  <div style={{ display: "flex", flexDirection: "column", gap: 8 }}>
    <a href="/kinda-act" style={subLinkStyle}>
      Kinda act：お見合いやデートに使いやすい場所
    </a>
    <a href="/kinda-glow" style={subLinkStyle}>
      Kinda glow：好きな人に会う前に、自分を整える時間
    </a>
  </div>
</div>
```

**補足：**

- 「この気持ちを、そのまま渡す」はクリップボードにコピーする機能（= カウンセラーへメモを渡す）。
- 「画像にして持っておく」は画像保存機能（= 自分のために残す）。
- 絵文字（📋、📷）は使わず、SVGアイコンで代替する。各ボタンのテキスト左側に、小さく subtle なSVGアイコンを配置してもよい（必須ではない）。

**ゲスト時のみ追加**（入会後ルートで、かつ未ログインの場合）：
```tsx
<button secondary-subtle>会員登録して履歴を残す</button>
```
※ 絵文字（💡）は使用しない。

#### 「他の機能も見てみる」サブ導線（v3 で追加）

活動中 5 ルート（waiting / omiai / date1 / kousai / multiple）の結果画面において、メイン 3 ボタンの直下に、Kinda act / Kinda glow への控えめなサブ導線を**全タイプ**で表示する。

**設計意図：**

- 活動中ユーザーは担当カウンセラーが既にいるため、Kinda type / talk への誘導は不要。
- ただし Kinda act（お見合い・デートで使いやすい場所）と Kinda glow（好きな人に会う前の美容）は、活動中こそ利用ニーズが高い。
- 「他の機能も見てみる」という控えめなトーンで、メイン CTA（カウンセラーに渡す／画像保存）を邪魔しない位置に置く。
- フォントサイズはメイン 3 ボタンより一段小さく、テキストリンク調にする。

**トーン規則：**

- ❌「次はここに行こう」「ぜひ使ってみて」（強い誘導）
- ✅「他の機能も見てみる」「ふたりの時間を、もっと心地よく」（控えめ・補助）
- カウンセラーに頼らず自分で動いて、という暗黙メッセージにならないよう注意。

**スタイル詳細（`subLinkStyle`）：**

```tsx
const subLinkStyle: React.CSSProperties = {
  display: "block",
  padding: "10px 16px",
  fontSize: 13,
  color: C.text,
  textDecoration: "none",
  border: `1px solid ${C.border}`,
  borderRadius: 8,
  transition: "background .2s",
};
```

ホバー時は背景を `C.cardLight` に変える程度の控えめな反応で十分。

**配置順序（活動中ルート共通）：**

```
[結果カード（summary・天気の解説・第1〜3層）]
[Kinda story 誘導カード（ポジティブ系3タイプ:
  omiai 晴れ間 / date1 朝焼け（淡）/ kousai 朝焼け のみ）]
[メイン3ボタン]
[★ サブ導線（Kinda act / Kinda glow）]
[ゲスト時のみ「会員登録して履歴を残す」]
```

**注意事項：**

- pre ルートには**追加しない**（既に 5 ボタン構成で type / talk / glow / story が揃っている）。
- ポジティブ系 3 タイプ（omiai 晴れ間 / date1 朝焼け（淡）/ kousai 朝焼け）では、既存の Kinda story 誘導カードの**さらに下**に配置する（メイン 3 ボタンの後）。
- 「Kinda act」「Kinda glow」のラベルは固定文言（上記の`a` タグ内）。タイプ別に文言を変えない。

## 画像保存機能

`html2canvas` を使用（`npm install html2canvas`）。

保存対象は結果画面全体ではなく、**引き継ぎ用の要約カード部分のみ**。新しく専用のコンポーネントを作ってレンダリングする：

```tsx
// components/ShareCard.tsx
// 800×1000px程度の縦長カード
// Kindaロゴ、結果タイプ、天気アイコン、主要な選択項目、日付を含む
// 自由記述は「表示する」オプションでON/OFF可能
```

保存ボタンクリック時：
1. オフスクリーンでShareCardをレンダリング
2. html2canvas で canvas に変換
3. `canvas.toDataURL('image/png')` で base64 化
4. ダウンロードリンクを生成してクリック

iPhoneの場合は長押し保存も案内する。

# localStorage 保存

Kinda note 完了時（`setScreen("result")` の直前）に保存：

```typescript
function saveKindaNoteHistory(route: string, resultType: string, weather: string, answers: object) {
  const history: KindaNoteHistoryItem[] = JSON.parse(
    localStorage.getItem("kinda_note_history") || "[]"
  );
  
  const newItem: KindaNoteHistoryItem = {
    id: crypto.randomUUID(),
    route,
    result_type: resultType,
    weather,
    answers,
    created_at: new Date().toISOString(),
  };
  
  history.push(newItem);
  
  // 最大100件、超えたら古いものから削除
  const trimmed = history.slice(-100);
  
  localStorage.setItem("kinda_note_history", JSON.stringify(trimmed));
}
```

フェーズ1では**保存するだけで、閲覧機能は作らない**。マイページの天気ダッシュボードはフェーズ5。

## 表示仕様

結果画面における天気解説の表示は以下のスタイルで行う。

```tsx
// 結果画面における天気解説の表示
<div className="weather-description" style={{
  marginTop: 16,
  marginBottom: 24,
  padding: "20px 24px",
  background: C.cardLight, // 既存トークン使用、淡い背景
  borderRadius: 16,
  fontFamily: '"Georgia", serif', // summary と同じ系統
  fontSize: 14,
  lineHeight: 1.9,
  color: C.text,
  letterSpacing: "0.02em",
}}>
  {weatherDescription}
</div>
```

- 第1層より少し小さめ
- summary の余韻を保つ詩的なタイポグラフィ
- 背景を淡くして「囲い込まれた詩」感を出す
- 改行は2文の間に1つ入れる（`\n` を `<br>` ではなく自然な行間として表現する。`white-space: pre-line` を併用してもよい）

  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()

-- RLS: 本人のみ閲覧可能、ゲスト(user_id null)は閲覧不可
CREATE POLICY "Users can view own sessions"
  ON kinda_note_sessions FOR SELECT
  USING (auth.uid() = user_id);

CREATE POLICY "Users can insert own sessions"
  ON kinda_note_sessions FOR INSERT
  WITH CHECK (auth.uid() = user_id OR user_id IS NULL);
