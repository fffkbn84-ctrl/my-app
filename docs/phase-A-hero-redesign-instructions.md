# Phase A：ヒーロー再設計 指示書（Claude Code 向け）

> **作成日**：2026-05-08
> **対象ブランチ**：既存の `implement~` 系ブランチ（**新規ブランチを作らないこと**）
> **スコープ**：トップページ（`/`）のヒーローセクションのみ。それ以外には触らない。
> **前提**：本書は Phase A 限定。Phase B（後半セクション補強）と Phase C（SEO・速度）は別タスク。

---

## 1. このタスクの目的

現在のヒーローは、ユーザーから以下のフィードバックを受けている：

1. なんかごちゃごちゃしている
2. 文字が多くて何を読めば良いか分かりにくい。文字色が白で読みにくい
3. 「いまの気持ちを整理する」をクリックする動機が弱い

**根本原因**：村のミニチュア画像の上に、メッセージとCTA2本が全部重なって乗っているため、画像と文字が干渉し、視線の優先順位が崩壊している。

**解決方針**：ビジュアルとテキストを **垂直に分離する**。画像は画像で完結させ、テキストとCTAは独立したセクションに置く。

## 2. 進め方の基本方針

**通常ケースは止まらず一気に実装→Vercelデプロイまで進めること**。確認のためにふうかさんに細かく問い合わせる必要はない。デプロイ後のスクリーンショットで判断する。

ただし、以下の **異常ケース** に該当した場合のみ、実装を止めてふうかさんに質問すること。

### 止まって質問すべき条件（これら以外は止まらない）

1. **ヒーローの実ファイルが特定できない**
   - `grep -rn "好きな人を見つけて" app/ components/ src/` で複数の候補が出て、どれが実ヒーローか判断できない
   - そもそも該当箇所が見つからない

2. **既存の `C` オブジェクトに必要な色トークンが見つからない**
   - 例：「濃ブラウン」「accentDark」相当の色が定義されていない
   - 勝手に新しいトークンを追加せず、既存値で代替できる場合は代替して進める。代替不能なら質問する

3. **ヒーロー以外のファイルにも変更が及びそう**
   - 想定差分はヒーロー本体（1〜2ファイル）と `page.tsx` のインポート構造のみ
   - それ以外（A' セクション、グローバルスタイル、共通コンポーネント等）に手を入れる必要が出たら、理由を添えて質問

4. **既存ヒーローの実装が指示書の前提と大幅に異なる**
   - 例：MDX で書かれている、Server Component と Client Component が複雑に絡んでいる、サードパーティの UI ライブラリで作られている等
   - 機械的なリファクタが効かず、構造から作り直す必要があるとき

### 通常ケースの進行

上記4条件に該当しなければ、以下の順で一気に進めること：

1. ヒーロー実ファイルを特定
2. セクション3の仕様に従って書き換え
3. ローカルでビルドエラーがないことを確認
4. `implement~` 系ブランチにコミット → push
5. Vercel デプロイ完了を確認
6. セクション5のチェックリストを実機（iPhone 16 Pro）で確認
7. 完了報告（セクション6のフォーマットで）

途中報告は不要。完了時に一括で報告すればよい。

## 4. 必ず守ること（重要原則）

### 4.1 既存のデザイントークンを尊重する
- `C` オブジェクト（colors/トークン）を新たに勝手に増やさない。既存の値で構築する。
- 必要な値が無い場合は、ふうかさんに確認を取ってから追加する。

### 4.2 既存フォント設定を変えない
- `Shippori Mincho`（ブランド・H1）、`Hiragino Kaku Gothic ProN`（本文）、`Georgia serif`（一部見出し）の使い分けを維持。

### 4.3 アニメーション・装飾を勝手に追加しない
- フェードインアニメーション、パララックス、ホバーエフェクトの追加は**指示書にない**ので追加禁止。
- glow shadow（ボタンの影）は仕様に含むので OK。

### 4.4 他セクションに触らない
- A' セクション（Kinda note 説明）、B セクション（4カード）、それ以降は **このタスクでは一切変更しない**。
- ヒーロー直下にあった「Kinda note セクション」が壊れないこと。

### 4.5 レスポンシブ
- モバイル最優先（iPhone 16 Pro 想定）。
- タブレット以上では Block 1〜3 を横並びにせず、**縦積みのまま中央寄せ・最大幅 480px** で固定して良い。
  - 理由：このサービスはモバイルが主戦場。デスクトップで「立派に見せる」必要は薄い。

### 4.6 アクセシビリティ
- ボタンと副CTAリンクには適切な `aria-label` または十分な視認可能テキストを。
- H1 は1ページに1つだけ（既存の他セクションに H1 が無いことを確認）。

# 付録：なぜこの設計なのか（Claude Code が判断に迷ったとき用）

## なぜ画像とテキストを分離するのか
モバイルで「画像にオーバーレイで文字を載せる」設計は、ほぼ必ず可読性が破綻する。理由は (1) 画像の明度が場所によって変わる、(2) ユーザーごとに端末の輝度が違う、(3) 黒オーバーレイを濃くすると今度は画像が死ぬ。**分離すれば、画像も文字も両方ベストな状態で見せられる**。

## なぜ主CTAは1つだけなのか
主CTAと副CTAを同じヒーロー内に並べると、ユーザーは「どっちが正解？」と迷い、結局**どちらも押さない**（決定回避）。Kinda note を「核」と決めた以上、主CTAは Kinda note 一択にして、副CTAは1段下に降ろす。

## なぜ「言葉にならないモヤモヤを、60秒で。」なのか
今のマイクロコピー「✓60秒で言葉になる」は **機能の説明**で、感情に訴えていない。ユーザーがクリックする動機は「機能」ではなく「自分の状態と一致する言葉」に出会えたとき。「言葉にならないモヤモヤ」は、ターゲット全員（pre / waiting / 活動中の全ルート）が抱えている共通の状態。

## なぜ副CTAをヒーロー外に出すのか
ヒーロー内に副CTAを置くと、ユーザーは「2つの動線」を同じ重みで見る。1段下に降ろすだけで「主CTAが本命、副は補助」という階層が成立する。
